LAB-02 : Amazon Virtual Private Cloud (VPC)¶

Amazon Virtual Private Cloud (Amazon VPC) vous permet de mettre en service une section du cloud AWS isolée de manière logique, au sein de laquelle vous pouvez lancer des ressources telles que des instances Amazon Elastic Compute Cloud (EC2) et Amazon Relational Database Service (RDS). Ce réseau virtuel ressemble beaucoup à un réseau traditionnel que vous pourriez implémenter dans votre propre centre de données, et présente l'avantage d'utiliser l'infrastructure évolutive d’AWS. Terminologie¶

[Suivant | Haut]

Dans cette section on va introduire quelques concepts clés liés au service VPC.

Virtual private cloud (VPC)¶

C'est un réseau virtuel dédié à votre compte AWS. Vous le créez et vous pourrez ensuite y ajouter des ressources. Vous pouvez spécifier une plage d'adresses IP pour votre VPC, créer des sous-réseaux, configurer des tables de routage et des passerelles. Vous pouvez aussi appliquer plusieurs couches de sécurité, à savoir les groupes de sécurité et les listes de contrôles d'accès au réseau (NACL), ce qui renforce le contrôle d'accès aux ressources dans chaque sous-réseau. Un VPC couvre toutes les zones de disponibilité de la région.

VPC par défaut¶

La première fois que vous lancer des ressources EC2, ces dernières sont créées dans un VPC par défaut (si aucun autre VPC personnalisé n'est pas précisé). En effet, chaque compte AWS est livré avec un VPC par défaut (172.31.0.0/16) qui a un sous-réseau public par défaut dans chaque zone de disponibilité. Il est connecté à une passerelle Internet et chaque instance lancée dans un sous-réseau par défaut possède une adresse IPv4 privée et une adresse IPv4 publique. Vous pouvez également créer votre propre VPC, appelé VPC personnalisé, et le configurer selon vos besoins.

Sous-réseau (subnet)¶

Après avoir créé un VPC, vous pouvez ajouter un ou plusieurs sous-réseaux. Un sous-réseau est une plage d'adresses IP dans votre VPC. Chaque sous-réseau doit résider entièrement dans une zone de disponibilité et ne peut pas s'étendre sur plusieurs zones.

Utilisez un sous-réseau public pour les ressources qui doivent être connectées à Internet et un sous-réseau privé pour les ressources qui ne doivent pas être connectées.

Table de routage (route table)¶

C'est l'ensemble de règles, appelées routes, qui permettent de déterminer où le trafic réseau est dirigé. Chaque sous-réseau de votre VPC doit être associé à une table de routage. Vous pouvez associer explicitement un sous-réseau à une table de routage personnalisée. Sinon, le sous-réseau est implicitement associé à la table de routage principale.

Un sous-réseau peut être associé à une seule table de routage à la fois, mais vous pouvez associer plusieurs sous-réseaux à une même table de routage.

Table de routage principale (main route table)¶

Il s'agit de la table de routage qui est associée automatiquement à votre VPC. Elle contrôle le routage pour tous les sous-réseaux qui ne sont associés explicitement à aucune autre table de routage. Par défaut, lorsque vous créez un VPC personnalisé, la table de routage principale contient seulement une route par défaut pour le routage local dans le VPC; elle permet aux instances dans le VPC de communiquer entre elles.

Passerelle Internet (Internet Gateways)¶

C'est la passerelle que vous attachez à votre VPC pour permettre la communication entre les ressources de votre VPC et l'internet. Elle est redondante et hautement disponible. Si un sous-réseau est associé à une table de routage comportant une route vers une passerelle Internet, il est considéré comme un sous-réseau public.

Ci-dessous la table de routage associée à un sous-réseau public. La première entrée est l'entrée par défaut pour le routage local dans le VPC. La deuxième entrée envoie tout le reste du trafic du sous-réseau à la passerelle Internet.

Passerelle NAT (NAT Gateways)¶

C'est un service de traduction d'adresses réseau (Network Address Translation). Elle est utilisée de façon à ce que les instances d'un sous-réseau privé puissent se connecter à des services en dehors de VPC, mais que des services externes ne puissent pas initier une connexion avec ces instances.

Lorsque vous créez une passerelle NAT, vous spécifiez l'un des types de connectivité suivants :

Publique (par défaut) : les instances dans des sous-réseaux privés peuvent se connecter à Internet via une passerelle NAT publique. Vous créez une passerelle NAT publique dans un sous-réseau public, et vous devez associer une adresse IP élastique à la passerelle NAT lors de sa création.

Privée : les instances dans des sous-réseaux privés peuvent se connecter à d'autres VPC ou à votre réseau local via une passerelle NAT privée.

Ci-dessous la table de routage associée au sous-réseau privé. La première entrée est l'entrée par défaut pour le routage local dans le VPC. La deuxième entrée envoie tout le reste du trafic du sous-réseau, tel le trafic lié à Internet, à la passerelle NAT.

Scénario d'un VPC avec deux sous-réseaux : public et privé¶

[Précédent | Suivant | Haut]

L'architecture de ce scénario comprend un VPC avec un sous-réseau public et un sous-réseau privé. Ce scénario est conseillé si vous voulez exécuter une application Web multi-tiers où vous allez placer :

  • les serveurs Web destinés au public dans un sous-réseau public
  • les serveurs de base de données qui ne doivent pas être accessibles par tout le monde dans un sous-réseau privé.

Les serveurs du sous-réseau public peuvent envoyer le trafic sortant directement sur le réseau Internet à travers une passerelle Internet, tandis que les serveurs du sous-réseau privé ne le peuvent pas. Ces derniers peuvent accéder à Internet (pour les mises à jour logicielles par exemple) à l'aide d'une passerelle de traduction d'adresses réseau (NAT) qui se trouve dans le sous-réseau public.

Vous pouvez définir la sécurité et le routage de sorte que les serveurs Web puissent communiquer avec les serveurs de base de données.

Création du VPC et de la passerelle Internet¶

Ouvrez la console du service VPC et sélectionnez "Your VPCs" à gauche puis cliquez sur "Create VPC". Donnez un nom à votre VPC et spécifier la valeur 10.0.0.0/16 pour le bloc d'adresse CIDR IPv4.

Activez l'accès à Internet en attachant une passerelle Internet à votre VPC. Pour ce faire :

  • Sélectionnez "Internet Gateways" dans le volet de navigation à gauche puis cliquez sur "Create internet gateway"
  • Spécifiez un nom à votre passerelle Internet et cliquez sur "Create internet gateway"
  • Cochez votre passerelle Internet dans la liste et à partir de "Actions" sélectionnez "Attach to VPC". Sélectionnez le VPC que vous avez créé puis cliquez sur "Attach internet gateway".

Création de sous-réseaux¶

Dans la barre de navigation gauche, sélectionnez "Subnets" puis cliquez sur "Create Subnet". Choisissez le VPC que vous avez créé puis créer deux sous-réseaux avec les informations suivantes:

Subnet name IPv4 CIDR block
sous-réseau 1 (public) my-public-subnet 10.0.1.0/24
sous-réseau 2 (privé) my-private-subnet 10.0.2.0/24

Création de la passerelle NAT¶

Vu que les adresses IP assignées aux instances d'un sous-réseau privé ne peuvent pas être utilisées en mode public, il est nécessaire d'utiliser une traduction d'adresses réseau (NAT) pour mapper les adresses IP privées avec une adresse publique pour les demandes, puis de remapper l'adresse IP publique avec les adresses privées pour la réponse.

Pour créer une passerelle NAT, procédez comme suit :

  • Sélectionnez "NAT Gateways" dans le volet de navigation à gauche puis cliquez sur "Create NAT gateway"
  • Spécifiez un nom à votre passerelle NAT
  • La passerelle NAT doit être créée dans le sous-réseau public avec un type de connectivité "Public"
  • Attribuez une adresse IP Elastic à votre passerelle NAT puis cliquez sur "Create Nat gateway"

Configuration de tables de routage¶

Autre que la table de routage principale vous pouvez créer des tables de routage personnalisées pour votre VPC. Pour créer une deuxième table de routage sélectionnez "Route tables" dans le volet de navigation à gauche puis cliquez sur "Create route table". Donnez un nom à votre table de routage et sélectionnez le VPC que vous avez créé.

Filtrer la liste des tables de routages avec le nom de votre VPC puis cochez votre table principale (ayant la valeur "Yes" dans la colonne "Main") puis sélectionnez "Routes". Vous allez constater la présence d'une seule entrée par défaut pour le routage local dans le VPC. Cliquez sur "Edit routes" et mettez à jour votre table de routage pour que 0.0.0.0/0 pointe vers l'identifiant de la passerelle NAT que vous avez créée.

De la même manière vous allez modifier votre table de routage personnalisée pour que 0.0.0.0/0 pointe vers l'identifiant de la passerelle Internet que vous avez associée à votre VPC.

Association de tables de routage¶

Dans la liste de vos sous-réseaux, sélectionnez le sous-réseau privé que vous avez créé (filtrez la liste avec le nom de votre VPC pour faciliter la recherche), puis sous l'onglet "Route table" vérifiez qu'il est associé à la table de routage principale.

Procéder de la même manière pour afficher la table de routage associée à votre sous-réseau public (par défaut, la table de routage principale). Cliquez sur "Edit route table association" puis sélectionnez la table de routage personnalisée que vous avez créée et enregistrez les modifications.

Rôle du hôte bastion¶

[Précédent | Haut]

Pour vous connecter à vos instances EC2 se trouvant à l'intérieur d'un sous-réseau privé, vous pouvez créer et configurer un hôte bastion (une instance EC2) dans le sous-réseau public qui va jouer le rôle d'un proxy.

Les hôtes bastion fournissent un accès sécurisé aux instances situées dans les sous-réseaux privés de votre VPC.

Pour créer un hôte bastion, vous pouvez lancez une nouvelle instance EC2 dans votre sous-réseau public ou utiliser l'instance que vous avez déjà créée dans ce sous-réseau.

Attribuez une adresse IP élastique à votre bastion et configurez vos groupes de sécurité comme suit :

  • Pour le hôte bastion, ajottez les règles de sécurité qui autorisent le trafic SSH entrant à partir des utilisateurs et le trafic SSH sortant vers le sous-réseau privé qui contient vos instances EC2.
  • Pour les instances EC2 dans le sous-réseau privé, ajoutez les règles de sécurité qui autorisent le trafic SSH entrant à partir du bastion.

Pour pouvoir se connecter à votre instance se trouvant dans le sous-réseau privée, le bastion doit avoir accès à sa clé privée (enregistré dans votre machine locale). Il existe deux méthodes pour ce faire :

  1. Méthode 1 : transférer le fichier clé vers votre instance bastion à partir de votre ordinateur Windows en utilisant l'outil WinSCP ou Linux en utilisant l'outil SCP.
  2. Méthode 2 (plus recommandée) : appliquer le principe de "SSH agent forwarding" qui consiste à utiliser un agent SSH qui stocke en mémoire les clés privées. Ainsi vous pouvez vous connecter au bastion à partir de votre machine locale et aux autres instances à partir du bastion sans y transférer la clé privée.

Pour les utilisateurs de Mac et Linux, procédez comme suit :

Ajoutez les clés privées de votre hôte bastion et de votre instance EC2 à l'agent SSH sur votre machine locale :

In [ ]:
$ ssh-add your-private-key.pem

Exécutez la commande suivante pour vérifier l'ajout des clés :

In [ ]:
$ ssh-add -L

Exécutez la commande suivante pour vous connecter à l'hôte bastion à l'aide de l'option -A (cette option permet le transfert de l'agent SSH) :

In [ ]:
$ ssh –A ec2-user@<bastion-IP-address or DNS-entry>

Une fois la connexion à l'hôte bastion est réussie, tapez la commande suivante dans le terminal du bastion pour vous connecter à votre instance EC2 :

In [ ]:
$ ssh ec2-user@<instance-IP-address or DNS-entry>

Notez bien que vous n'avez pas besoin de fournir explicitement une clé dans les deux commandes précédentes. L'agent SSH essaie séquentiellement toutes les clés qu'il possède jusqu'à ce qu'il trouve la bonne clé.

Pour les utilisateurs de Windows, suivez les étapes suivantes :

  1. Dans votre machine locale, cherchez et ouvrez le service "Pageant"
  2. Allez à la zone de notification et cliquez avec le bouton droit sur "Pageant" pour y ajouter vos clés privées
  3. Procédez à une connexion SSH avec l'outil PuTTY. Saisissez l'adresse IP de votre instance dans la case "Host Name (or IP address)". Dans le volet de navigation à gauche, sélectionnez "auth" puis cochez "Allow agent forwarding" (vous n'avez plus besoin de fournir votre clé privée).
  4. Une fois la connexion à l'hôte bastion est réussie, tapez la commande suivante dans le terminal du bastion pour vous connecter à votre instance EC2 :
In [ ]:
$ ssh ec2-user@<instance-IP-address or DNS-entry>

Bonnes pratiques :

  • Créez une instance EC2 dédiée pour jouer le rôle d'hôte bastion exclusivement.
  • Créez plusieurs bastions dans différentes zones de disponibilité (où vous avez des instances EC2).
  • Configurer le groupe de sécurité du bastion pour autoriser les connexions SSH uniquement à partir des adresses IP connues et de confiance.
  • Configurez les groupes de sécurité des instances pour accepter les connexions SSH uniquement à partir d'un bastion.
  • Utilisez le "SSH agent forwarding" au lieu de placez vos clés privées dans le bastion.
