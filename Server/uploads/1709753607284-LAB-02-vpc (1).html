Virtual private cloud (VPC)

C'est un rseau virtuel ddi votre compte AWS. Vous le crez et vous pourrez ensuite y ajouter des ressources. Vous pouvez spcifier une plage d'adresses IP pour votre VPC, crer des sous-rseaux, configurer des tables de routage et des passerelles. VPC par dfaut

La premire fois que vous lancer des ressources EC2, ces dernires sont cres dans un VPC par dfaut (si aucun autre VPC personnalis n'est pas prcis). En effet, chaque compte AWS est livr avec un VPC par dfaut (172.31.0.0/16) qui a un sous-rseau public par dfaut dans chaque zone de disponibilit. Il est connect une passerelle Internet et chaque instance lance dans un sous-rseau par dfaut possde une adresse IPv4 prive et une adresse IPv4 publique. Vous pouvez galement crer votre propre VPC, appel VPC personnalis, et le configurer selon vos besoins.

Sous-rseau (subnet)

Aprs avoir cr un VPC, vous pouvez ajouter un ou plusieurs sous-rseaux. Un sous-rseau est une plage d'adresses IP dans votre VPC. Chaque sous-rseau doit rsider entirement dans une zone de disponibilit et ne peut pas s'tendre sur plusieurs zones.

Utilisez un sous-rseau public pour les ressources qui doivent tre connectes Internet et un sous-rseau priv pour les ressources qui ne doivent pas tre connectes.

Table de routage (route table)

C'est l'ensemble de rgles, appeles routes, qui permettent de dterminer o le trafic rseau est dirig. Chaque sous-rseau de votre VPC doit tre associ une table de routage. Vous pouvez associer explicitement un sous-rseau une table de routage personnalise. Sinon, le sous-rseau est implicitement associ la table de routage principale.

Un sous-rseau peut tre associ une seule table de routage la fois, mais vous pouvez associer plusieurs sous-rseaux une mme table de routage.

Table de routage principale (main route table)

Il s'agit de la table de routage qui est associe automatiquement votre VPC. Elle contrle le routage pour tous les sous-rseaux qui ne sont associs explicitement aucune autre table de routage. Par dfaut, lorsque vous crez un VPC personnalis, la table de routage principale contient seulement une route par dfaut pour le routage local dans le VPC; elle permet aux instances dans le VPC de communiquer entre elles.

Passerelle Internet (Internet Gateways)

C'est la passerelle que vous attachez votre VPC pour permettre la communication entre les ressources de votre VPC et l'internet. Elle est redondante et hautement disponible. Si un sous-rseau est associ une table de routage comportant une route vers une passerelle Internet, il est considr comme un sous-rseau public.

Ci-dessous la table de routage associe un sous-rseau public. La premire entre est l'entre par dfaut pour le routage local dans le VPC. La deuxime entre envoie tout le reste du trafic du sous-rseau la passerelle Internet.

Passerelle NAT (NAT Gateways)

C'est un service de traduction d'adresses rseau (Network Address Translation). Elle est utilise de faon ce que les instances d'un sous-rseau priv puissent se connecter des services en dehors de VPC, mais que des services externes ne puissent pas initier une connexion avec ces instances.

Lorsque vous crez une passerelle NAT, vous spcifiez l'un des types de connectivit suivants :

Publique (par dfaut) : les instances dans des sous-rseaux privs peuvent se connecter Internet via une passerelle NAT publique. Vous crez une passerelle NAT publique dans un sous-rseau public, et vous devez associer une adresse IP lastique la passerelle NAT lors de sa cration.

Prive : les instances dans des sous-rseaux privs peuvent se connecter d'autres VPC ou votre rseau local via une passerelle NAT prive.

Ci-dessous la table de routage associe au sous-rseau priv. La premire entre est l'entre par dfaut pour le routage local dans le VPC. La deuxime entre envoie tout le reste du trafic du sous-rseau, tel le trafic li Internet, la passerelle NAT.

Cration du VPC et de la passerelle Internet

Ouvrez la console du service VPC et slectionnez "Your VPCs" gauche puis cliquez sur "Create VPC". Donnez un nom votre VPC et spcifier la valeur 10.0.0.0/16 pour le bloc d'adresse CIDR IPv4.

Activez l'accs Internet en attachant une passerelle Internet votre VPC. Pour ce faire :

  • Slectionnez "Internet Gateways" dans le volet de navigation gauche puis cliquez sur "Create internet gateway"

  • Cochez votre passerelle Internet dans la liste et partir de "Actions" slectionnez "Attach to VPC". Slectionnez le VPC que vous avez cr puis cliquez sur "Attach internet gateway".

Dans la barre de navigation gauche, slectionnez "Subnets" puis cliquez sur "Create Subnet". Choisissez le VPC que vous avez cr puis crer deux sous-rseaux avec les informations suivantes:

Cration de la passerelle NAT

Vu que les adresses IP assignes aux instances d'un sous-rseau priv ne peuvent pas tre utilises en mode public, il est ncessaire d'utiliser une traduction d'adresses rseau (NAT) pour mapper les adresses IP prives avec une adresse publique pour les demandes, puis de remapper l'adresse IP publique avec les adresses prives pour la rponse.

Pour crer une passerelle NAT, procdez comme suit :

  • Slectionnez "NAT Gateways" dans le volet de navigation gauche puis cliquez sur "Create NAT gateway"
  • Spcifiez un nom votre passerelle NAT
  • La passerelle NAT doit tre cre dans le sous-rseau public avec un type de connectivit "Public"
  • Attribuez une adresse IP Elastic votre passerelle NAT puis cliquez sur "Create Nat gateway"

Configuration de tables de routage

Autre que la table de routage principale vous pouvez crer des tables de routage personnalises pour votre VPC. Pour crer une deuxime table de routage slectionnez "Route tables" dans le volet de navigation gauche puis cliquez sur "Create route table". Donnez un nom votre table de routage et slectionnez le VPC que vous avez cr.

Filtrer la liste des tables de routages avec le nom de votre VPC puis cochez votre table principale (ayant la valeur "Yes" dans la colonne "Main") puis slectionnez "Routes". Vous allez constater la prsence d'une seule entre par dfaut pour le routage local dans le VPC. De la mme manire vous allez modifier votre table de routage personnalise pour que 0.0.0.0/0 pointe vers l'identifiant de la passerelle Internet que vous avez associe votre VPC.

Rle du hte bastion

[Prcdent | Haut]

Pour vous connecter vos instances EC2 se trouvant l'intrieur d'un sous-rseau priv, vous pouvez crer et configurer un hte bastion (une instance EC2) dans le sous-rseau public qui va jouer le rle d'un proxy.

Les htes bastion fournissent un accs scuris aux instances situes dans les sous-rseaux privs de votre VPC.

Pour crer un hte bastion, vous pouvez lancez une nouvelle instance EC2 dans votre sous-rseau public ou utiliser l'instance que vous avez dj cre dans ce sous-rseau.

Attribuez une adresse IP lastique votre bastion et configurez vos groupes de scurit comme suit :

  • Pour le hte bastion, ajottez les rgles de scurit qui autorisent le trafic SSH entrant partir des utilisateurs et le trafic SSH sortant vers le sous-rseau priv qui contient vos instances EC2.
  • Pour les instances EC2 dans le sous-rseau priv, ajoutez les rgles de scurit qui autorisent le trafic SSH entrant partir du bastion.

Pour pouvoir se connecter votre instance se trouvant dans le sous-rseau prive, le bastion doit avoir accs sa cl prive (enregistr dans votre machine locale). Il existe deux mthodes pour ce faire :

  1. Mthode 1 : transfrer le fichier cl vers votre instance bastion partir de votre ordinateur Windows en utilisant l'outil WinSCP ou Linux en utilisant l'outil SCP.
  2. Mthode 2 (plus recommande) : appliquer le principe de "SSH agent forwarding" qui consiste utiliser un agent SSH qui stocke en mmoire les cls prives. Ainsi vous pouvez vous connecter au bastion partir de votre machine locale et aux autres instances partir du bastion sans y transfrer la cl prive.

Pour les utilisateurs de Mac et Linux, procdez comme suit :

Ajoutez les cls prives de votre hte bastion et de votre instance EC2 l'agent SSH sur votre machine locale :

In [ ]:
$ ssh-add your-private-key.pem

Excutez la commande suivante pour vrifier l'ajout des cls :

In [ ]:
$ ssh-add -L

Excutez la commande suivante pour vous connecter l'hte bastion l'aide de l'option -A (cette option permet le transfert de l'agent SSH) :

In [ ]:
$ ssh A ec2-user@<bastion-IP-address or DNS-entry>

Une fois la connexion l'hte bastion est russie, tapez la commande suivante dans le terminal du bastion pour vous connecter votre instance EC2 :

In [ ]:
$ ssh ec2-user@<instance-IP-address or DNS-entry>

Notez bien que vous n'avez pas besoin de fournir explicitement une cl dans les deux commandes prcdentes. L'agent SSH essaie squentiellement toutes les cls qu'il possde jusqu' ce qu'il trouve la bonne cl.

Pour les utilisateurs de Windows, suivez les tapes suivantes :

  1. Dans votre machine locale, cherchez et ouvrez le service "Pageant"
  2. Allez la zone de notification et cliquez avec le bouton droit sur "Pageant" pour y ajouter vos cls prives
  3. Procdez une connexion SSH avec l'outil PuTTY. Saisissez l'adresse IP de votre instance dans la case "Host Name (or IP address)". Dans le volet de navigation gauche, slectionnez "auth" puis cochez "Allow agent forwarding" (vous n'avez plus besoin de fournir votre cl prive).
  4. Une fois la connexion l'hte bastion est russie, tapez la commande suivante dans le terminal du bastion pour vous connecter votre instance EC2 :
In [ ]:
$ ssh ec2-user@<instance-IP-address or DNS-entry>

Bonnes pratiques :

  • Crez une instance EC2 ddie pour jouer le rle d'hte bastion exclusivement.
  • Crez plusieurs bastions dans diffrentes zones de disponibilit (o vous avez des instances EC2).
  • Configurer le groupe de scurit du bastion pour autoriser les connexions SSH uniquement partir des adresses IP connues et de confiance.
  • Configurez les groupes de scurit des instances pour accepter les connexions SSH uniquement partir d'un bastion.
  • Utilisez le "SSH agent forwarding" au lieu de placez vos cls prives dans le bastion.
